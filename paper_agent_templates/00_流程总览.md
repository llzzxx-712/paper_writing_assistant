# 论文多Agent通用流程总览

适用目标：让任意用户仅凭 Rule_list 与本模板包，即可驱动经理agent组织模块agent与审查agent协作产出规范论文草稿。

## 1. 角色定义

- 用户：提供项目背景、现有代码与素材，做最终裁决。
- 经理agent：流程编排者，负责蓝图、术语表、任务下发、收敛决策。
- 模块agent：按章节或子模块写作，产出正文与配套报告。
- 审查agent：按规则卡审查并输出可执行返修指令。

## 2. 标准流水线

1. 启动阶段：经理agent读取 Rule_list 与模板包，确认章节分工、时间轮次、验收标准。
2. 蓝图阶段：经理agent生成论文蓝图、术语表、主张-证据映射表。
3. 首轮写作：经理agent向各模块agent下发章节任务prompt，模块agent提交四件套。
4. 审查返修：经理agent向审查agent下发审查prompt，审查agent按 rule_id 生成问题单与返修prompt。
5. 多轮收敛：模块agent按返修prompt定点修改，经理agent评估是否进入下一轮。
6. 收尾阶段：核心章节达标后，经理agent撰写摘要与第5章总结展望，做全局一致性审计。

## 3. 模块agent标准提交物

每个模块agent必须提交四件套：
- 正文草稿（章节正文）
- 自检报告（逐条对照规则卡）
- 证据清单（结论到图表/实验/代码/数据映射）
- 待确认项（需要用户或经理决策的问题）

建议文件组织：
- 独立四文件更稳，便于多轮追踪。
- 若追求轻量，可用单文件四区块，但后续需严格固定区块标题。

## 4. 数据或图表缺失时的处理

若模块agent发现某段结论缺乏数据支撑，不允许硬写结论，应执行：
1. 标记阻塞类型为 evidence_gap。
2. 提出可执行的数据补全方案（脚本思路、输入、输出、验证指标）。
3. 请求用户确认后，先生成数据/图，再回填论文段落。

## 5. 自定义细化需求的接入方式

规则卡无法覆盖全部细节时，采用“增量规则注入”机制：
1. 用户提出细化要求。
2. 经理agent判断影响范围。
3. 经理agent将新增要求写成临时规则（custom_rule）。
4. 在本轮下发给相关模块agent与审查agent。
5. 若该规则具有通用性，再回写到 Rule_list。

## 6. 通过标准

建议每轮按以下门槛判断是否通过：
- 必过：must/forbidden 无 open 问题。
- 可放行：should 项少量未完成且不影响结论。
- 禁止放行：证据不足、章节口径冲突、摘要与正文结论不一致。
